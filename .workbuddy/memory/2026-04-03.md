## 微博发布失败 bug 修复 — XSRF-TOKEN 缺失

### 现象
- Bot 执行 `weibo/post` 命令发布微博时返回 403 "invalid csrf token"
- 扩展显示"已同步"，数据库凭据 status=active，cookie 包含 SUB
- 浏览器中 weibo.com 正常登录

### 根因
- `routes/cli.js` 中 weibo/post 和 weibo/profile 的请求头**缺少 `X-XSRF-TOKEN`**
- 微博 AJAX 接口（weibo.com/ajax/*）要求 CSRF token 验证
- cookie 中有 `XSRF-TOKEN` cookie，但请求时没有通过 header 传回
- 同样的问题也影响了凭据验证端点（verify）的微博部分

### 修复
- 新增 `weiboHeaders(cookies)` 辅助函数，自动从 cookie 数组提取 `XSRF-TOKEN` 并设置 `X-XSRF-TOKEN` header
- 修复 `weibo/profile`、`weibo/post` 命令的请求头
- 修复凭据验证 `POST /api/cli/credentials/verify/:id` 中微博验证逻辑的请求头
- 增加了 `Origin`、`Accept` 等必要 header 使请求更像浏览器

### 验证
- 修复后测试发布成功：`{"ok":1, "id":"5283727322121134"}`
- 测试微博已删除

## WorkerClaw v0.16.0 — 私有虾消息处理重构

### 问题
- 私有虾收到主人私信时，回复"你可以发个任务给我"（打工虾逻辑）
- 私有虾收到外部人员私信时，没有拒绝

### 设计
- **`mode: 'private'` 私有虾只认主人**（`rentalState.renterId`，从服务器拉取）
- **主人发私信** → 直接用 AgentEngine 执行指令，不走任务流
- **外部人员发私信** → 礼貌拒绝"我是主人专属内勤虾"
- **聊天室消息** → 私有虾完全不参与
- **ownerId 安全设计**: 不可本地配置，仅从服务器 `checkRentalStatus()` 拉取 `renterId`
  - 防止出租者篡改主人身份冒用浏览器登录态发微博等

### 修改文件
- `src/task/task-manager.ts`:
  - 新增 `isPrivateMode()`, `isOwner()`, `handleOwnerDirectMessage()`, `handlePrivateExternalMessage()`
  - `handlePrivateMessageReply()`: 私有虾模式分流（主人→直接执行 / 外部→拒绝）
  - `handleChatMessageReply()`: 私有虾直接 return 不参与聊天室
- `src/core/config.ts`: 无改动（ownerId 不加入本地配置）
- `src/core/workerclaw.ts`:
  - `isPrivateMode()` 判断扩展：`config.mode === 'private'` OR `rentalState.active`
  - 被租用时自动停止 behaviorScheduler，租赁到期后自动恢复（仅限原 public 虾）

## v0.16.6 — LLM 限速自动切换端点

### 问题
- Agent 执行任务时 LLM 遇到 429 限速，重试 3 次都打同一个端点，直接失败
- 根因：`getCurrentConfig()` 在重试循环外只调用一次，3 次 retry 共享同一个 endpoint

### 修复
- `LLMClient.chat`：每次重试都调用 `getCurrentConfig(failedApiKey)` 获取新端点
- `LLMKeyRotator.getNextEndpoint(skipApiKey)`：支持跳过指定端点
- `LLMKeyRotator.recordFailure`：429 错误触发 60s 冷却，冷却期内自动跳过
- 日志增加 `attempt` 字段，方便追踪重试和端点切换

### 同时修复 v0.16.5 的 config 传递问题
- `TaskManagerConfig` 接口新增 `mode?` 和 `ownerId?`
- `WorkerClaw` 创建 TaskManager 时传入 `config.mode` 和 `(config as any).ownerId`
