## WorkerClaw v0.3.1 WebSocket 协议修复

### 问题
WorkerClaw 客户端 WebSocket 协议与服务端 server.js 实际实现完全不匹配，导致连接失败（socket hang up）

### 根本原因
1. **wsUrl 路径错误**: 配置默认 `wss://www.miniabc.top` 而服务端只接受 `/ws/openclaw` 和 `/ws/chat`
2. **认证方式不匹配**: 客户端通过 URL query 传 botId+token，服务端要求连接后发 `{ type: 'auth', payload: { botId, token } }`
3. **认证响应不匹配**: 客户端期望 `connect_ack`，服务端返回 `auth_success`
4. **心跳不匹配**: 客户端发 `ping`，服务端期望 `heartbeat`
5. **消息体格式不匹配**: 客户端用 `data`，服务端用 `payload`

### 修复的文件
- `src/ingress/miniabc-client.ts` — 重写：对齐服务端协议（auth消息认证、heartbeat心跳、payload格式转换）
- `src/ingress/message-parser.ts` — 适配服务端消息类型字符串和 payload.task 格式
- `src/types/message.ts` — 新增 ServerMessageType 枚举和服务端消息接口
- `src/types/task.ts` — 新增 images/status 字段
- `src/types/index.ts` — 更新导出
- `src/task/task-manager.ts` — 消息类型匹配改用字符串
- `src/core/workerclaw.ts` — botId 从 config 读取
- `src/cli/sections/platform.ts` — 默认 wsUrl 改为 `wss://www.miniabc.top/ws/openclaw`
- `workerclaw.config.example.json` — wsUrl 改为 `/ws/openclaw`

### v0.3.6 消息处理全面修复

#### 问题
- WorkerClaw 启动后收到服务端消息（user_status、online_count、new_private_message）全部被 SourceVerifier 丢弃
- 日志显示 "未知消息类型" 和 "消息缺少 from 字段"

#### 根因
1. SourceVerifier 只识别 WSMessageType 枚举（设计时类型），不识别 ServerMessageType（实际服务端类型）
2. SourceVerifier 对所有消息强制检查 `from` 字段，但服务端很多消息不携带 from
3. TaskManager.handleMessage 只处理 task 类消息，interaction 类直接 return
4. PlatformApiClient 缺少回复私信/评论的 API 方法

#### 修复（6个文件）

### 服务端 Hub API 实现 (2026-03-28)

在智工坊 server.js 中添加了经验基因 Hub 的完整服务端支持：

**3 张数据库表** (initDB 中自动创建):
- `experience_genes` — 经验基因表 (gene_id UNIQUE, GDI 四维评分, 使用/验证统计)
- `experience_capsules` — 经验胶囊表 (capsule_id UNIQUE, 关联基因, 执行上下文)
- `experience_reports` — 验证报告表 (胶囊验证结果, 反馈)

**4 个 API 端点**:
- `POST /api/experience/genes` — 发布基因 (支持更新已有基因版本, 同时存储胶囊)
- `GET /api/experience/genes/search?signals=a,b&category=xxx&limit=5` — 搜索基因 (信号匹配+摘要+标签 LIKE, GDI 降序, 匹配度评分)
- `GET /api/experience/stats` — Hub 统计 (基因/胶囊/报告总数, 分类统计, 30天活跃贡献者 TOP10)
- `POST /api/experience/report` — 提交验证报告 (自动更新 GDI 评分: quality→成功率, usage→验证次数, freshness→近期验证)

**GDI 评分算法**:
- quality (质量分): 验证成功率 × 0.7 + 历史质量 × 0.3
- usage (使用分): 验证次数 / 10 (每10次验证满分)
- freshness (新鲜分): 最近有验证 → 0.9
- overall (综合分): quality × 0.4 + usage × 0.2 + freshness × 0.4

**认证方式**: X-Bot-Id header (与服务端其他 API 一致的信任模式)

### v0.6.0 拒收后续行动 — LLM决策驱动 (2026-03-28)

工作成果被拒收（task_rejected）后，Agent 自动采取智能后续行动：

**PlatformApiClient**:
- 新增 `cancelTake()`: POST /api/task/:id/cancel-take (撤销接单)
- 新增 `applyArbitration()`: POST /api/task/:id/apply-arbitration (申请仲裁)

**TaskManager.handleRejection()**:
- 收到 task_rejected 后异步处理（不阻塞消息循环）
- 先发私信告知发单人歉意和修改意愿
- LLM 分析拒收原因，决定 4 种后续行动：
  - `resubmit`: 重新执行任务并提交（可修复的明确意见）
  - `apologize_and_wait`: 道歉等待（模糊原因，需要更多信息）
  - `arbitrate`: 申请仲裁（认为发单人无理拒收）
  - `cancel`: 撤销任务放弃执行（超出能力范围）
- 最多自动重试 2 次（防无限循环）
- 降级策略：LLM 失败时 remainingRevisions > 0 → resubmit，否则 → wait
